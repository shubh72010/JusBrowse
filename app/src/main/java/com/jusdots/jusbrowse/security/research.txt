On our Android 14 devices, Chrome 120+ is the active browser engine, and “Android System WebView” is effectively the same Chromium build (v120 or newer). Key points:

    Update mechanism: On older Android (below 5.0) WebView could only be updated with an OS release. On modern OS, Chrome/Chromium is updated rapidly via Play services. Pixel 8/7a on Android 14 are running Chromium 120+ out of the box, and WebView is the same version.

    User-agent and features: The User-Agent and built-in features differ slightly between Chrome/Chromium and WebView builds. Recent Android WebView inherits Chromium’s defaults (including privacy changes) but some flags may differ. For instance, Chrome’s Privacy Sandbox has reduced the UA string (see below) which also applies in WebView 120. Older versions (e.g. WebView on Android 6) lacked these changes. Locking to Chromium 120 ensures the latest WebView fingerprint surface.

In practice, this means our test environment should use the Android System WebView APK and Chrome that correspond to version 120. (For example, Pixel 8 Pro updated to Android 14 will include WebView 120.x.) This keeps WebView’s JavaScript and rendering identical to Chrome 120, so any fingerprint differences truly come from OS or hardware, not mismatched browser versions.

Hardware and firmware are tightly coupled to the OS/driver combo. Pixel models have unique SoCs, GPUs, sensors and security chips, and each Android release carries the corresponding drivers and firmware. For our scenario:

    SoC/GPU: Pixel 8 Pro is built on Google Tensor G3 (4 nm) with an Arm Immortalis-G715 GPU. Pixel 7a/7 Pro use Tensor G2 with a Mali-G710 GPU. These differences can change graphics and WebGL fingerprints (vendor/renderer strings, supported extensions, GPU driver bugs, etc.).     Memory/Storage: The Pixel 8 Pro supports up to 12 GB RAM and UFS 3.1 storage, similar to Pixel 7a/7, but new RAM may have different timing characteristics. Pixel drivers for memory manage this.

    Security Co-processor: Both Pixel 8/8 Pro and 7a use Titan M2. Titan firmware is OS-specific. For example, the Android 14 build includes the latest Titan M2 binary, which can affect secure enclave operations.

    Biometrics/Sensors: Pixel 8 Pro and 7a use in-display optical fingerprint readers (7a inherited the 7 Pro’s under-display sensor). Camera drivers (e.g. 50 MP main sensor) also updated per OS. All sensor firmware (ambient light, gyroscope, temperature sensor on Pixel 8 Pro, etc.) are tied to that Pixel’s Android build.

Browser Fingerprinting Surface Variations

Modern browsers expose many “fingerprinting surfaces” (canvas, WebGL, audio, fonts, timezone, language, device memory, etc.). Crucially, these surfaces have been evolving across browser versions to reduce tracking. Even minor version changes in Chromium or WebView can alter default values. For Android 14/Chromium 120 we note:

    User-Agent & Client Hints: Chrome’s Privacy Sandbox now sends a reduced UA string on all requests. Navigator interfaces (navigator.userAgent, etc.) are similarly truncated. The full device info is only available via explicit Client Hints (UA-CH) if the server requests it. In our environment, Chrome 120/Android 14 will use the reduced UA by default. This contrasts with older Chrome/Android builds which included full model/version in the UA. (WebView 120 will follow suit, sending the trimmed UA.)

    Graphics (Canvas/WebGL): The canvas and WebGL fingerprint depend on GPU vendor, driver, and browser. We expect Chrome 120 on Pixel (Immortalis GPU) to report something like “ANGLE (Intel, AMD, or SwiftShader)” and a specific renderer string. Firefox or older Chrome versions might report differently. Also, browser updates may randomize or “hash” certain canvas bits to prevent trivial fingerprinting. For example, browsers often tamper with toDataURL() noise. Major browsers (Chrome, Safari, Firefox) have begun limiting high-entropy graphics attributes (added noise, fixed color precision, etc.). Chrome’s WebGL or 2D canvas outputs on v120 may not exactly match those from v119 or v121, so we should use the exact version.

    Fonts & Audio: Chrome’s canvas fingerprint often includes system font enumeration (via measuring text). Firefox 145+ now blocks access to local font lists and only exposes “standard” fonts, adding noise. Chrome’s current builds also apply fingerprint mitigation to audio context data. In practice, we should assume Chromium 120 exposes fewer unique font/audio signals than older Chrome. WebView (Chromium) on Android 14 will have a similar font set as Chrome, but any differences (e.g. WebView might not load the same locale fonts by default) could cause discrepancies.

    Device Characteristics: Properties like navigator.platform, language, screen resolution, hardwareConcurrency, etc. are sometimes fuzzed. For example, Chrome reduces navigator.platform detail, and Firefox fixes hardwareConcurrency=2 and truncates screen height. In WebView on Android, most of these are inherited from the OS. On Pixel 8/14, navigator.platform will likely report just “Linux armv8l” or similar. Note that each browser engine (Chrome vs WebView vs Firefox) may handle these slightly differently.

    Privacy Protections: Overall, “major browsers limit or randomize key attributes” to fight fingerprinting. Chrome’s Privacy Sandbox and Firefox’s ETP (Enhanced Tracking Protection) mean that a fingerprint collected on Android 14/Chrome 120 may be less unique than on older setups. For thoroughness, we should compare Chrome/WebView vs other engines (e.g. Firefox 145+ on Android 14) to see the delta. Firefox 145 in Private mode, for instance, now blocks query selectors for fonts and normalizes many APIs.

In summary, fingerprinting surfaces can differ between Chromium versions and between Chromium vs Firefox. Some specific examples: Chrome 120 will use the reduced UA and may cap float precision in canvas; Firefox 145 adds noise to images and hides local fonts. System WebView 120 inherits Chrome’s engine changes, so its fingerprint should closely match Chrome 120 on Android 14. Any testing on Android 14 should therefore use the exact Chrome/WebView build intended (120+). This ensures that the evaluated fingerprint (user agent, WebGL strings, etc.) matches what real users on that OS would generate.

Android OS & Kernel Differences

Android versions differ in how they sandbox and spawn apps. All apps are forked from the Zygote process, and starting in Android 8 (Oreo) Google added a seccomp‐BPF filter at Zygote startup to block unused syscalls for security. In Android 14 the kernel must support seccomp-BPF with thread synchronization (TSYNC) to meet AOSP requirements. For example, Android O’s security team blocked 17 syscalls (like swapon) at the kernel level for all apps. These low-level differences (e.g. which syscalls are allowed) can subtly change what APIs or devices are visible to apps and thus affect what fingerprinting data can be read.

    Zygote & seccomp: Android’s Zygote now installs a seccomp filter by default. This means app processes inherit a sanitized syscall set, which can vary slightly by OS patch. (A recent CVE-2024-31317 shows how complex this is: a flaw in ZygoteProcess.java allowed an app to escalate privileges through unsafe deserialization.) In short, each Android release can tweak which syscalls or capabilities apps can use, so Android 14’s sandbox differs from Android 13 or earlier.

    TEE (Trusted Execution) handling: Pixel devices include a Titan M security co-processor (“TEE” or TrustZone). The Pixel 8/7a use Titan M2, which works closely with the OS for secure functions like credential storage. Firmware updates to Titan M2 often come bundled with OS updates. In practice, this means Android 14 on a Pixel 8/8 Pro has an updated Titan M2/Tensor G3 security stack not present on older Pixels.
